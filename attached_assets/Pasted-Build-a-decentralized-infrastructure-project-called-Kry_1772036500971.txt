Build a decentralized infrastructure project called "Krydo".

Krydo is a privacy-preserving financial trust system where users can prove financial credibility without revealing sensitive financial information.

The system should be designed as a serious startup-grade infrastructure prototype, not a simple student project.

You are free to design the architecture and choose technologies yourself, but the system must follow the functional requirements below.

CORE IDEA

Krydo should implement a hierarchical trust model similar to a central banking system:

Root Authority → Issuers → Users → Verifiers

The system should run on blockchain as much as practical.

AUTHORITY MODEL
Root Authority

The wallet address that deploys the main contract automatically becomes the Root Authority.

Root Authority acts like a central authority similar to a regulator.

Root Authority must be able to grant and revoke issuer privileges.

Root Authority permissions must be enforced by the blockchain.

Issuers

Issuers represent trusted institutions.

Issuers must:

Be approved by the Root Authority

Have wallet-based identity

Be able to issue credentials to users

Be able to revoke credentials

Issuers must NOT:

Grant issuer permissions

Modify authority structure

Users

Users are normal participants.

Users must:

Log in using wallet addresses

Own credentials

Request credentials from issuers

Generate proofs or verifications when needed

Wallet address should be the only identity mechanism.

No password authentication.

Verifiers

Verifiers represent applications or services that need to verify users.

Verifiers must be able to:

Verify user credentials

Check issuer authenticity

Check credential validity

Verifiers should not require special permissions.

Verification should be publicly accessible.

CREDENTIAL MODEL

The system must use structured credentials.

Credentials must include:

Issuer identity

User identity

Claim data

Timestamp

Cryptographic authenticity

Credentials must preserve privacy.

Sensitive financial data must never be stored publicly.

The blockchain should only store minimal necessary information such as references or hashes.

BLOCKCHAIN REQUIREMENTS

The blockchain layer must:

Store trusted issuer identities

Store credential references

Store credential status

Enforce authority permissions

The blockchain must NOT:

Store sensitive financial data

Perform heavy computations unnecessarily

FUNCTIONAL REQUIREMENTS

The system must support:

1 Root Authority deployment

2 Adding trusted issuers

3 Removing trusted issuers

4 Listing issuers

5 Issuing credentials

6 Revoking credentials

7 Viewing credential status

8 Verifying credentials

USER EXPERIENCE

The system should include an interface that allows:

Root Authority:

Manage issuers

Issuers:

Issue credentials

Revoke credentials

Users:

Connect wallet

Request credentials

View credentials

Verifiers:

Verify credentials

SECURITY REQUIREMENTS

The system must:

Enforce permissions on blockchain

Prevent unauthorized issuance

Prevent fake issuers

Prevent credential tampering

DESIGN GOALS

The system should:

Look like early-stage startup infrastructure

Have clean architecture

Separate authority logic from credential logic

Be modular

Be scalable

OUTPUT EXPECTED

Produce a complete working prototype including:

Smart contracts

Backend logic if needed

Frontend interface

Deployment instructions

Explain the architecture decisions clearly.
